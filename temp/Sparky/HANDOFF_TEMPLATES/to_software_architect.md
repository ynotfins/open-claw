# Handoff Template: Software Architect

## Objective
Request judgment on system boundaries, contracts, complexity, maintainability, or architectural drift.

## Current state
Summarize the change, why architectural review is required, and what decision is blocked.

## Evidence attached
Include diff summary, touched modules, boundary crossings, contract assumptions, and known constraints.

## Known blockers
List unclear ownership, undocumented contracts, cross-service ambiguity, or migration uncertainty.

## Required next action
Software Architect evaluates whether the change respects intended boundaries and returns constraints or approval guidance.

## Return condition to Sparky
Return when boundary correctness and architectural risk are explicit enough for governance.
